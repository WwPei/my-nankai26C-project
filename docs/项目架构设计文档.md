# Emoji Dungeon 阶段0架构设计文档

## 1. 目标

本阶段将仓库从 Qt Quick 模板切换为 Qt 6 Widgets 项目骨架，并建立后续肉鸽地牢射击游戏可扩展的基础结构。当前阶段只提供：

- Widgets 启动框架与页面占位
- `QStackedWidget` 页面流转骨架
- `QGraphicsView` 游戏主场景占位
- 数据配置结构与静态配置表
- `GameFactory` 工厂接口与创建入口
- Data-View 分离接口声明

本阶段不包含具体战斗、刷怪、碰撞伤害、武器发射与升级结算逻辑。

## 3. 架构决策

### 3.1 渲染

- 采用 `QGraphicsView + QGraphicsScene` 作为游戏主渲染容器。
- 理由：该方案天然适配大量动态对象、碰撞检测与场景管理，优于用普通 `QWidget` 手工绘制游戏对象。

### 3.2 页面管理

- 采用 `QStackedWidget` 管理开始页、职业选择页、主游戏页、升级页。
- 理由：阶段0需要先稳定页面切换骨架，后续 UI 迭代时可以保持页面职责清晰。

### 3.3 数据与视图分离

- 敌人与子弹采用 `Data + View` 双类结构：
  - `EnemyData` / `EnemyView`
  - `BulletData` / `BulletView`
- 理由：数据状态与图形显示分离后，便于单元测试、逻辑复用、后续更换表现层。

### 3.4 对象创建

- 所有对象统一通过 `GameFactory` 创建。
- 理由：武器、敌人、特性都基于配置 ID 初始化，工厂可以屏蔽创建细节并集中配置依赖。

### 3.5 配置管理

- 所有游戏配置放在 `game_data.h` 中静态定义，不在运行期读取外部文件。
- 理由：阶段0优先稳定接口与构建链路，避免过早引入配置加载系统。

### 3.6 内存管理

- `QObject` 派生对象使用 Qt 父子关系托管。
- `QGraphicsItem` 相关对象由 `QGraphicsScene` 管理移除，后续销毁策略统一为“先从场景解绑，再延迟清理关联 QObject”。
- 理由：避免在场景仍持有对象时直接删除，降低悬空指针与泄漏风险。

## 4. 模块职责

### 4.1 `EmojiDungeonWindow`

- 管理应用主窗口
- 负责页面创建、页面切换、导航信号连接
- 不承载具体战斗逻辑

### 4.2 `ClassSelectPage`

- 展示职业占位按钮
- 发出 `classSelected(PlayerClassId)` 信号
- 不直接创建玩家对象

### 4.3 `GameMainPage`

- 持有 `QGraphicsScene` 与 `QGraphicsView`
- 作为游戏视图容器
- 当前仅展示场景占位，不运行游戏循环

### 4.4 `UpgradePage`

- 展示特性按钮占位
- 发出特性选择与返回主页面信号
- 后续升级结算逻辑在逻辑模块实现

### 4.5 `GameFactory`

- 提供配置查询接口
- 提供玩家、武器、敌人、子弹、特性的统一创建入口
- 当前阶段返回空指针作为占位，实现“接口先行”

### 4.6 核心领域接口

- `Player`：玩家数据与行为接口
- `Weapon`：武器行为接口
- `Trait`：特性应用接口
- `EnemyData` / `EnemyView`：敌人数据与视图接口
- `BulletData` / `BulletView`：子弹数据与视图接口
- `GameView`：所有游戏图形对象的统一视图基类

## 5. 信号槽约定

- 页面导航信号：
  - `ClassSelectPage::classSelected`
  - `ClassSelectPage::backRequested`
  - `GameMainPage::upgradeRequested`
  - `GameMainPage::exitRequested`
  - `UpgradePage::traitSelected`
  - `UpgradePage::resumeRequested`

- 逻辑模块预留信号：
  - `Player::healthChanged`
  - `Player::weaponChanged`
  - `EnemyData::positionChanged`
  - `EnemyData::defeated`
  - `BulletData::expired`
  - `GameView::removalRequested`

设计原则是“页面只负责导航，游戏对象只负责状态变化通知”，避免跨模块直接强耦合调用。

## 6. 后续阶段建议

- 补齐 `Player`、`Weapon`、`EnemyData`、`BulletData` 的最小实现类

- 接入碰撞检测和伤害结算

- 引入更完整的资源与美术占位管理
