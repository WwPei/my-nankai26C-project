## 三、课题要求

### 3.1 问题描述

随着 Rogue-lite 类游戏（如《吸血鬼幸存者》《Hades》）在独立游戏市场的持续走红，"短局高密度决策"的游戏范式受到广泛关注。这类游戏通常需要玩家在有限时间内，通过职业选择、升级决策和操作技巧来抵御不断增长的外界压力。

作为 C++ 课程大作业，本项目目标为：**从零构建一款完整可运行、架构清晰、具备可扩展性的俯视角射击生存游戏**，完整实践面向对象程序设计、多态、工厂模式、MV 分离架构等核心 C++ 知识点。

### 3.2 核心问题

设计并实现一个具备以下特征的交互式游戏系统：

1. **实时战斗系统**：支持玩家移动、射击、冲刺，与多类型敌人实时交互
2. **动态难度体系**：随波次递增，敌人血量、伤害按指数增长（HP × 1.05^wave，DMG × 1.03^wave）
3. **数据驱动的游戏配置**：所有职业、武器、敌人、特性的数值通过静态配置表管理，分离逻辑与数据
4. **完整的升级决策链路**：击败敌人获取经验 → 升级，随机获得 5 种特性/武器强化或特殊子弹选择
5. **多 AI 行为体系**：6 种差异化 AI 行为模式（追逐、射击游击、蓄力冲锋、自爆、Boss 多阶段等）
6. **工程化架构**：MV 分离、工厂模式、信号槽解耦、CMake 多目标构建

## 四、主要流程

### 4.1 整体流程

#### 4.1.1 设计思路

项目从一开始就确立了"先骨架、后血肉、再打磨"的渐进式开发思路：

**阶段 0（架构先行）**：
- 在写任何游戏逻辑之前，先建立 8 个纯虚抽象接口和 GameFactory 工厂
- 确定 QGraphicsView + QGraphicsScene 渲染方案和 QStackedWidget 页面管理方案
- 确立"页面只负责导航，游戏对象只负责状态变化通知"的信号槽约定
- 所有接口方法标 `[[nodiscard]]`，编译期约束返回值不可忽略

**阶段 1-2（功能实现）**：
- 从接口出发，逐步实现各 Basic* 类的具体逻辑
- 建立 16ms 主循环驱动全游戏帧更新
- 按照"数据驱动"原则，所有游戏数值从 GameConfig 配置表读取，不在代码中硬编码

**阶段 3-4（打磨优化）**：
- UI 层面从功能优先转向体验优先（悬浮 HUD、动画弹窗、动态表情）
- 数值层面降低经验门槛、提高怪物密度，让游戏节奏更快更爽
- 视听层面接入完整 BGM 和音效系统

**层级依赖规则（严格遵守）**：
- 类型一被所有其他类型依赖，不依赖任何类型
- 类型二的实现由类型三完成（`final` 封闭继承）
- 类型四的模型绑定通过类型二的接口指针完成（依赖倒置原则）
- 类型五通过类型六的子系统操作类型四的 View 对象，通过类型二接口操作类型三的 Model 对象
- 类型七是类型三的子类型（SpecialBulletData 继承 BasicBulletData）

#### 4.1.4 MV 分离设计

项目最核心的架构原则是 **Model-View 分离**：

```
数据层 (Model) — 派生自 QObject:
    Player (抽象)     → BasicPlayer (final)
    Weapon (抽象)     → BasicWeapon (final)
    EnemyData (抽象)  → BasicEnemyData (final)
    BulletData (抽象) → BasicBulletData → SpecialBulletData (final)
    Trait (抽象)      → BasicTrait (final)

视图层 (View) — 派生自 QGraphicsObject:
    GameView (抽象)
      ├── EnemyView (抽象)    → BasicEnemyView (final)
      └── BulletView (抽象)   → BasicBulletView (final)
                              → SpecialBulletView (final)
```

View 层通过 `syncFromData()` 纯虚方法每帧从 Model 拉取最新状态，不持有 Model 的数据拷贝，仅持有 `QPointer<ModelInterface>` 安全引用。

#### 4.1.5 数据流架构

```
QTimer(16ms) 主循环（GameMainPage::handleBattleTick）
    │
    ├─→ WaveManager::advanceFrame        ─→ 经验累计/波次推进/升级触发
    ├─→ EnemyDirector::spawnTestEnemy    ─→ 按波次系列生成敌人
    ├─→ EnemyData::updateAI(delta, pos)  ─→ 6 种 AI 状态机决策
    ├─→ Player::tickMovement()           ─→ WASD 位移 + 边界限制
    ├─→ Weapon::advanceCooldown()        ─→ CD 递减 + 开火信号
    ├─→ CombatCoordinator::handleFire    ─→ 子弹创建（平行/扇形/特殊）
    ├─→ BulletData::advanceFrame         ─→ 子弹飞行 + 过期判定
    ├─→ EnemyData::advanceFrame          ─→ 敌人移动 + 射击/冲锋
    ├─→ resolveCombatCollisions          ─→ 碰撞检测 + 伤害结算
    ├─→ cleanupExpiredBullets            ─→ 清理过期子弹
    ├─→ cleanupDefeatedEnemies           ─→ 清理击败敌人 + 经验结算
    └─→ refreshAllViews                  ─→ syncFromData() 更新渲染
```

### 4.2 算法或公式

#### 4.2.1 敌人 AI 行为算法

6 种 AI 行为通过 BasicEnemyData 中的 `switch-case` 统一处理，核心是以"与玩家距离"为决策依据的有限状态机。

**Chase（追逐）—— SkeletonNew 使用**：
```
每帧: setTargetPosition(playerPosition) → 直线移动
```

**KeepDistance（保持距离）—— 预留，未分配**：
```
if (与玩家距离 < keepDistanceMin):
    朝远离玩家方向移动
else:
    向玩家靠近到 keepDistanceMin 距离
```

**Charge（蓄力冲锋）—— Ogre、XenoBeast 使用，三段流程**：
```
Phase 1 — 追逐: 向玩家移动
Phase 2 — 蓄力: CD 到达后原地不动 chargeUpMs（Ogre: 800ms / XenoBeast: 500ms）
Phase 3 — 冲锋: 以 velocity × chargeSpeedMult 冲向玩家 chargeDurationMs
Phase 4 — 冷却: 进入 chargeCooldownMs 冷却（Ogre: 3s / XenoBeast: 2.5s）
```

**ShootAndMove（射击游击）—— Jester、Ghost、Robot 使用**：
```
if (与玩家距离 >= keepDistanceMin × 85%):
    向玩家移动
elif (距离 < keepDistanceMin × 85%):
    朝远离玩家方向移动
if (shootCooldown ≤ 0):
    向玩家方向发射 shootBulletCount 发子弹，重置 CD
```

**SuicideBomb（自爆）—— 预留，未分配**：
```
始终向玩家移动，距离 ≤ suicideRadius 时引爆造成范围伤害
```

**Boss（多阶段首领）—— DemonLord、BoneLord、UFO、AlienPilot 使用**：
```
Phase 1: 始终向玩家移动 + CD 到达时发射圆弧弹幕（shootBulletCount 发子弹）
Phase 2（HP < 50%）: 移动速度 ×1.5，射击 CD 重置
spawnCooldown ≤ 0 时召唤当前系列的小兵
```

**动态难度公式**：
```
enemyHpActual  = baseHp × 1.05^wave
enemyDmgActual = baseDmg × 1.03^wave
expMultiplier   = 1.0  (wave 1-3)
                = 1.1  (wave 4-7)
                = 1.2  (wave 8-9)
                = 1.5  (wave 10+)
```

#### 4.2.2 武器射击算法

```
输入: weaponId, origin（发射位置）, direction（朝向）

1. 查 WeaponConfig 获取 projectileCount 和 bulletStyle
2. 根据 bulletStyle 创建普通弹或特殊弹（SpecialBulletData）
3. 并行弹道计算:
   - PeaShooter（豆豆发射器）: 使用平行偏移 kParallelOffset = 10px
   - SpreadBlaster（散射喷发器）: 使用旋转扩散 kSpreadStepDegrees = 12°
   - ArcWand（弧光法杖）: 单发直线
4. extraProjectiles（武器强化）: 在基础弹道两侧追加额外子弹
5. 注入武器强化参数: pierceCount, bulletSizeScale, damageMultiplier, comboMultiplier
6. 场景添加 BulletView → 推入 bullets 向量
```

#### 4.2.3 碰撞检测算法

```
resolveCombatCollisions():
  // 玩家子弹 vs 敌人
  for each playerBullet:
      collidingItems = bullet.view → collidingItems()
      for each collided:
          if (item is EnemyView):
              if (trait == Frostbite): enemy → applySlow(40%, 2s)
              if (trait == CriticalStrike && random() < 15%): damage ×= 1.5
              enemy → receiveDamage(damage)
              if (pierceCount > 0): pierceCount-- (continue)
              else: mark bullet expired (break)

  // 敌人子弹 vs 玩家
  for each enemyBullet:
      if collide with playerMarker:
          player → receiveDamage(damage)

  // 近战接触伤害
  maxContactDamage = max(enemy.contactDamage)
  if (maxContactDamage > 0 && playerDamageCooldown ≤ 0):
      player → receiveDamage(maxContactDamage)
      playerDamageCooldown = 600ms
```

#### 4.2.4 经验与等级系统算法

```
经验阈值公式: threshold[n] = Σ(10 × i²), i = 1..n-1
第 n 级升级需要: 10 × (n-1)² 经验值

升级判定: levelForExperience(exp) → 二分查找阈值表
经验获得: enemyBaseExp × waveExpMultiplier × playerExpMultiplier
升级触发: 从 kAllUpgradeOptions 随机抽取 5 个（已拥有则自动升级）
特性叠加: QuickHands Lv2 → damage × 1.15², speed × 1.05²（乘法叠加）
```

#### 4.2.5 特殊子弹特效算法

**TrackAndAttach（追踪挂身）—— 哈基米追踪猫**：
```
发射后每帧: 锁定最近敌人 → 调整飞行方向指向目标
命中后: 挂载到目标身上 → 每帧对被挂目标造成 baseDmg × 0.15 持续伤害 + 15% 减速
持续 3 秒后自动销毁
```

**Stun（眩晕）—— "不是哥们！？"**：
```
命中后: 目标 entersStunState(1.5 秒)
眩晕中: AI update 跳过（不移动/不攻击），View 渲染蓝色覆盖层
```

**KnockbackWithBonus（击退+撞墙奖励）—— "带派不老铁"**：
```
命中后: 计算子弹飞行方向向量 × 150 像素 = 击退目标位置
if (击退目标位置超出场景边界):
    将目标位置钳制到边界
    额外造成 totalDamage × 0.30 的撞墙奖励伤害
```

#### 4.2.6 核心技术选型

| 技术决策 | 选择方案 | 理由 |
|----------|----------|------|
| 渲染引擎 | QGraphicsView + QGraphicsScene | 天然适配大量动态对象、碰撞检测、多图层管理 |
| 页面管理 | QStackedWidget（3 页） | 页面职责清晰，切换简洁 |
| 升级 UI | 悬浮弹窗 + QVariantAnimation | 重叠于战斗画面，避免整页切换撕裂感 |
| 对象创建 | GameFactory 工厂模式 | 统一入口，基于配置 ID 实例化 |
| 碰撞检测 | collidingItems() + HP 比较双重校验 | 确保碰撞可靠性 |
| 主循环 | QTimer(16ms) ≈ 60FPS | 固定帧率驱动，保证游戏节奏统一 |
| 音频方案 | QMediaPlayer + QAudioOutput | 兼容 FFmpeg MP3 解码 |
| 配置管理 | 编译期静态表（`inline const`） | 零运行时开销，所有数据编译期确定 |

### 4.3 开发过程中遇到的难点及解决方法

#### 难点一：QSoundEffect MP3 解码不兼容

**问题**：使用 `QSoundEffect` 播放 `.mp3` 命中音效时，控制台持续报 `Error decoding source`，音效无法播放。

**根因**：Qt 6.10 的 FFmpeg 多媒体后端与 `QSoundEffect` 的 MP3 解码管道不兼容，`QSoundEffect` 内部使用独立解码路径，仅稳定支持 `.wav` 格式。

**解决方法**：彻底移除 `QSoundEffect`，统一改用 `QMediaPlayer` + `QAudioOutput`。两者使用同一 FFmpeg 后端，MP3 兼容稳定。同时简化了音效管理（无需分别维护两套音频对象）。

```cpp
// 修改前（不兼容）
QSoundEffect *hitSound = new QSoundEffect(this);
hitSound->setSource(QUrl("qrc:/musics/hit.mp3"));  // Error decoding source

// 修改后（兼容）
QMediaPlayer *hitPlayer = new QMediaPlayer(this);
QAudioOutput *audioOutput = new QAudioOutput(this);
hitPlayer->setAudioOutput(audioOutput);
hitPlayer->setSource(QUrl("qrc:/musics/hit.mp3"));
```

#### 难点二：升级弹窗渲染崩溃

**问题**：战斗页面切换到升级时程序崩溃，堆栈指向 `QGraphicsOpacityEffect::draw()`。

**根因**：`QGraphicsOpacityEffect` 与 `QGraphicsView` 的 GPU 渲染管道冲突。`QGraphicsView` 内部使用独立的绘制路径，`QGraphicsOpacityEffect` 尝试在 QWidget 层级重定向绘制目标时与 QGraphicsView 的 viewport 绘制发生冲突。

**解决方法**：用 `QVariantAnimation` + QSS `background: rgba(r, g, b, alpha)` 动态改变 `alpha` 通道值实现淡入淡出效果。动画从 `rgba(10,12,18,0)` 过渡到 `rgba(10,12,18,0.88)`（淡入 350ms OutCubic），反过来淡出 280ms InCubic。

```cpp
// 废弃方案（会崩溃）
QGraphicsOpacityEffect *effect = new QGraphicsOpacityEffect(this);
upgradeOverlay->setGraphicsEffect(effect);

// 最终方案（稳定）
QVariantAnimation *fadeIn = new QVariantAnimation(this);
fadeIn->setDuration(350);
fadeIn->setEasingCurve(QEasingCurve::OutCubic);
fadeIn->setStartValue(0);
fadeIn->setEndValue(0.88);
connect(fadeIn, &QVariantAnimation::valueChanged, this, [this](const QVariant &v) {
    int alpha = qRound(v.toDouble() * 255);
    upgradeOverlay->setStyleSheet(
        QString("background: rgba(10,12,18,%1);").arg(alpha));
});
```

#### 难点三：升级弹窗事件捕获冲突

**问题**：使用 `qApp->installEventFilter(this)` 全局捕获点击事件实现"点击弹窗外部关闭"时，升级弹窗内部按钮点击无响应。

**根因**：全局事件过滤器优先级高于子控件事件分发，会将所有鼠标事件在分发到子控件之前拦截。

**解决方法**：删除 `qApp` 事件过滤器，改为在每个升级卡片上 `installEventFilter`，通过 `property("cardIndex")` 独立捕获点击。弹窗外部点击通过遮罩层的 `mousePressEvent` 处理。

#### 难点四：角色/子弹拖尾残影

**问题**：游戏运行中角色移动和子弹飞行出现拖尾残影，画面脏区域未正确重绘。

**根因**：`QGraphicsView` 默认使用 `MinimalViewportUpdate` 模式，只重绘被标记为脏的矩形区域。在高密度对象（22 个敌人 + 大量子弹同时存在）场景下，增量脏区域标记不够精确，导致上一帧的部分像素未被覆盖。

**解决方法**：设置 `setViewportUpdateMode(QGraphicsView::FullViewportUpdate)`，改为每帧全视口重绘。虽然理论上性能开销更大，但实测在 60FPS 下帧率稳定，且彻底解决了残影问题。

```cpp
// battle_arena_view.cpp 构造函数中
setViewportUpdateMode(QGraphicsView::FullViewportUpdate);  // 单行修复
```

#### 难点五：WASD 在升级界面仍然生效

**问题**：升级弹窗显示时，玩家 WASD 移动仍然响应，导致角色在弹窗背后移动。

**根因**：旧流程先 `setBattleActive(false)` 停止战斗循环，但未先设置 `BattleFlowState::Upgrade`。键盘事件处理函数中判断 `battleState != BattleFlowState::Active` 时才拦截，由于状态未更新，WASD 继续生效。

**解决方法**：改为先调用 `setBattleState(BattleFlowState::Upgrade)` 再 `setBattleActive(false)`，确保键盘事件按 `battleState` 正确拦截。

```cpp
// 修改前
setBattleActive(false);  // 停止战斗循环
// battleState 仍为 Active → WASD 继续响应

// 修改后
setBattleState(BattleFlowState::Upgrade);  // 先设置状态
setBattleActive(false);  // 再停止循环
```

#### 难点六：CMake + windres 路径空格编译失败

**问题**：项目路径 `F:\QT projects\emoji_dungeon_v3` 含空格，CMake 将 C++ include 目录路径传递给 `windres.exe` 编译 `.rc` 资源文件时，空格被误解析为参数分隔符。

**根因**：CMake 内部将 `target_include_directories` 的所有路径作为 `-I` 参数传递给 windres，但 windres 使用空格作为参数分隔，无法正确处理含空格的路径。

**解决方法**：手动先用 `windres` 编译 `.rc` → `.o`，再通过 CMake 的 `target_link_libraries` 直接链接预编译对象文件，绕过 CMake 的 include 路径传递。

```cmake
# CMakeLists.txt 中的绕行方案
add_custom_command(
    OUTPUT app_icon_res.o
    COMMAND windres -i app_icon.rc -o app_icon_res.o
    DEPENDS app_icon.rc
)
target_link_libraries(emoji_dungeon PRIVATE "${CMAKE_CURRENT_SOURCE_DIR}/app_icon_res.o")
```

#### 难点七：Boss 血条每帧抖动

**问题**：Boss 血条跟随 Boss 场景坐标移动时出现抖动。

**根因**：每帧追踪 Boss 的 `scenePos()` 并映射到 viewport 坐标，在高帧率下场景坐标更新和 UI 重绘之间存在一帧延迟，导致血条位置在相邻帧之间微小跳动。

**解决方法**：Boss 血条改为固定在波次进度条正下方（`positionBossBarFixed((w - 300) / 2, kHudMargin + 28)`），不再追踪 Boss 场景坐标。Boss 血条始终在屏幕顶部居中，视觉效果更稳定。

---

#### 难点八：背景图片不显示

**问题**：BattleArenaView 上设置的背景图片无法显示。

**根因**：背景图片在 `paintEvent` 中绘制，但 `QGraphicsView` 的子控件（viewport）覆盖了 `paintEvent` 的绘制结果。

**解决方法**：将背景图片绘制移至 `BattleArenaView::drawBackground()`，由 QGraphicsView 的 background 机制处理，确保背景在所有图形项之后、viewport 之前渲染。

---

#### 难点九：机器人敌人不射击

**问题**：Robot（机器人）敌人不会发射子弹。

**根因**：`shootTriggerDistance` 字段被误设为 `true`（布尔值隐式转换为 1.0），与期望的射击触发距离语义不符，导致射击判定逻辑错误。

**解决方法**：将 `shootTriggerDistance` 改为 `0.0F`，使机器人在任意距离都能触发射击。

## 六、收获

### 6.1 面向对象设计与多态

通过本项目深刻理解了抽象基类和多态的实际应用价值。项目定义了 8 个纯虚抽象接口（Player、Weapon、EnemyData、BulletData、Trait、GameView、BulletView、EnemyView），所有游戏实体都通过基类指针进行操作，这样在添加新敌人类型或新子弹类型时无需修改战斗协调器等核心逻辑。

```cpp
// 多态的核心价值：操作接口，不关心具体类型
void CombatCoordinator::handleWeaponFireRequested(const QPointF &origin, const QPointF &direction) {
    // m_playerWeapon 是 Weapon* 抽象指针，实际可能是 PeaShooter/SpreadBlaster/ArcWand
    BulletConfig config = m_factory->bulletConfigFor(m_playerWeapon->bulletStyle());
    // 统一通过接口操作，不关心具体武器类型
}
```

### 6.2 虚析构函数的重要性

项目中所有抽象基类都声明了虚析构函数。这确保了通过基类指针删除子类对象时，子类的析构函数能被正确调用，避免内存泄漏。

```cpp
// player.h
class Player : public QObject {
    Q_OBJECT
public:
    virtual ~Player() = default;  // 虚析构函数，确保子类正确析构
    // ...
};
```

### 6.3 MV 分离架构的价值

Model（数据）和 View（渲染）的彻底分离是项目中最重要的架构决策。最初 V1 版本将数据和渲染逻辑混在一起，导致代码难以维护。V2 重构后实现了完整的 MV 分离：

- Model 层只关心数据状态变化（位置、血量、CD），不涉及任何渲染代码
- View 层只关心如何绘制（贴图、血条、特效），通过 `syncFromData()` 从 Model 拉取状态
- 两者通过信号槽通信，Model 发出状态变化信号，View 或其他系统响应

这种分离使得：
1. 可以独立更换渲染方案（从 QPainter 纯色绘制到 PNG 贴图渲染，Model 代码零修改）
2. 可以独立测试数据逻辑（不依赖图形环境）
3. 代码职责清晰，单个文件行数控制在合理范围

### 6.4 工厂模式解决对象创建复杂性

GameFactory 工厂类集中管理所有游戏对象的创建，通过配置 ID 统一实例化。这使得：

- 添加新敌人/武器/特性只需在配置表中增加条目，工厂自动支持
- 对象创建逻辑集中在一处，避免各处散布构造函数调用
- 工厂返回接口指针，调用方不依赖具体实现类

### 6.5 编译期类型安全

项目中充分利用了 C++17 的编译期特性来防止运行时错误：

- `[[nodiscard]]`：标记所有不应忽略返回值的函数，编译器在返回值被忽略时产生警告
- `enum class`：13 个强类型枚举，避免隐式类型转换和命名空间污染
- `final`：所有 `Basic*` 实现类标记为 `final`，防止意外的继承扩展
- `inline const`：所有配置表编译期确定，零运行时开销

### 6.6 增量式开发的价值

项目经历了 V1→V2→V3→V4 四个版本，每次迭代都建立在上一版本稳定的基础上：

- V1 验证了技术可行性（Qt Graphics View + 游戏主循环）
- V2 在稳定架构上大幅扩充内容（从 3 种敌人到 10 种，新增 12 种特殊子弹）
- V3 在不改变逻辑的前提下工程化重组（10 个子目录，CMake Strategy A）
- V4 在内容完整的基础上打磨体验（悬浮 HUD、弹窗动画、BGM 系统）

这种渐进式开发避免了"大爆炸"式集成的风险，每次交付都有一个可运行、可演示的版本。

### 6.7 Git 版本管理的体会

V1→V2→V3→V4 四个版本均有完整的 CHANGELOG 记录和架构文档。每次提交都有明确的版本标识，能够清晰追溯每个功能的引入时间。当 V4 出现 Bug 时，可以通过 Git 历史快速定位问题是在哪个版本引入的，极大提高了调试效率。
